# Dashboard Intake Trend Executive Redesign

Date: 2026-04-13
Owner: Codex for Kenneth
Status: Draft for review

## Objective

Upgrade the dashboard's featured intake-trend module from a basic heading-plus-chart panel into an executive summary card that communicates the current intake posture in under two seconds.

The redesign must:

- keep the existing dashboard API contract unchanged
- derive executive metrics from the existing 12-month intake series in the browser
- preserve the dashboard's current command-center visual language
- improve readability on both desktop and mobile

## Current State

The intake trend currently renders as:

- a section heading in `views/dashboard/index.php`
- a single chart canvas with no supporting summary metrics
- a generic line-chart treatment generated by `public/assets/js/dashboard.js`

This makes the chart readable, but not immediately useful for executive scanning because the user must interpret the trend line without a concise summary of what changed.

## Scope

In scope:

- redesign the intake-trend card structure in `views/dashboard/index.php`
- add client-side executive summary derivation in `public/assets/js/dashboard.js`
- restyle the featured intake card in `public/assets/css/dashboard.css`
- add or update narrow tests for the intake-trend shell and summary logic where practical

Out of scope:

- backend route, controller, or service changes
- new dashboard API fields
- changes to other dashboard cards except for incidental spacing consistency
- introducing new frontend dependencies

## Design Direction

The module should read as an executive briefing card, not a raw analytics widget.

Tone:

- premium
- calm
- fast to scan
- operationally credible

The chart remains important, but the card should lead with the latest intake number and supporting context rather than with the graph itself.

## Proposed Experience

### 1. Card Structure

The featured intake card will contain four layers:

1. Heading block
2. Executive stat rail
3. Insight sentence
4. Chart stage

The heading block keeps the existing category framing:

- eyebrow label: `Twelve-month intake`
- title: `Intake Trend`

The supporting copy should be tightened to match the executive treatment and sit above the metrics rather than compete with the chart.

### 2. Executive Stat Rail

Add a summary rail above the chart with three items:

- `Latest intake`
  - primary visual anchor
  - large numeral for the most recent month value
  - smaller caption for the month label
- `vs previous month`
  - delta chip showing absolute change and percent change
  - up/down/flat visual state
- `12-month peak`
  - secondary benchmark showing peak month and peak value

This rail should communicate the most useful answer first: what happened most recently, and how that compares to last month.

### 3. Insight Sentence

Add one plain-language insight line under the stat rail. This line is generated from the same derived metrics and should translate the chart into a short narrative.

Examples:

- `Intake accelerated this month, approaching the annual high.`
- `Intake softened this month, easing below last month's pace.`
- `Intake held steady month over month.`
- `Intake data is available, but no recent variation stands out.`

The insight must stay short and avoid implying operational risk that the available data does not prove.

### 4. Chart Treatment

The line chart remains below the summary content, but becomes calmer and more polished:

- lighter gridlines
- softer area fill under the line
- stronger emphasis on the active or terminal point
- cleaner tooltip surface
- reduced visual noise from chart chrome

The chart should visually support the stat rail, not fight it for attention.

## Data Derivation

All executive metrics will be derived in the browser from the existing `payload.charts.intake` series.

Expected source shape remains the current Chart.js-friendly structure:

```js
{
  labels: ['May', 'Jun', 'Jul', ...],
  datasets: [
    {
      label: 'Intake',
      data: [12, 9, 14, ...]
    }
  ]
}
```

Derived model requirements:

- latest month label
- latest month total
- previous month total, when available
- absolute change
- percent change
- direction: `up`, `down`, or `flat`
- 12-month peak label
- 12-month peak total
- plain-language insight
- a validity flag indicating whether the series contains usable values

The derivation logic should be isolated in a dedicated helper function so it can be tested separately from chart mounting.

## Empty and Sparse Data Handling

The module must degrade cleanly when data is incomplete.

Rules:

- if there is no intake dataset, show a calm empty summary and skip misleading deltas
- if only one month has data, show the latest total and hide month-over-month comparison
- if all values are zero, show neutral language instead of a positive or negative trend claim
- if labels and data lengths mismatch, fail gracefully and prefer the empty summary path

The UI should never show fabricated percentages or directional language when the comparison basis is missing.

## Visual System

The intake card should stay aligned with the dashboard's existing premium surfaces, but use a more deliberate hierarchy than the generic chart panels.

Recommended visual treatment:

- featured panel keeps the elevated surface and border language already used on the dashboard
- subtle radial or ambient highlight behind the stat rail
- oversized primary numeral with restrained supporting labels
- compact delta pill with three states:
  - cool blue for increase or healthy momentum
  - warm amber for decrease
  - neutral styling for flat or unavailable comparison
- chart area separated with breathable spacing rather than a hard divider

The result should feel executive, not flashy.

## Responsive Behavior

Desktop:

- heading and supporting copy remain left-aligned
- stat rail spans horizontally in three compact blocks
- chart sits beneath with balanced breathing room

Tablet and mobile:

- stat rail stacks into a tighter grid or vertical rhythm
- latest intake remains visually dominant
- peak and delta items compress without truncating key numbers
- chart height scales down without collapsing readability

The mobile version should preserve the summary-first hierarchy rather than shrinking everything evenly.

## Accessibility

Requirements:

- maintain readable contrast for stat text, pills, and chart details in both themes
- do not encode delta meaning with color alone; include text or symbols
- ensure insight text remains in normal DOM content, not only within canvas
- keep the existing semantic heading hierarchy intact

## Implementation Plan Preview

Expected file touch points:

- `views/dashboard/index.php`
  - add intake summary shell above the chart canvas
  - add placeholder containers for derived insight content
- `public/assets/js/dashboard.js`
  - add helper to derive executive intake summary from the existing monthly series
  - render stat rail and insight content into the new shell
  - refine intake chart configuration without changing the API request flow
- `public/assets/css/dashboard.css`
  - add intake-specific featured-card styling
  - add stat rail, delta pill, insight line, and responsive rules
- tests
  - add or update a dashboard view assertion for the new shell
  - add a narrow JS test if an existing pattern supports isolated dashboard helper tests

## Validation Plan

Minimum validation:

- narrow PHP view test confirming the intake-trend summary shell renders
- narrow JS verification for summary derivation logic if the repository already supports lightweight frontend logic testing
- manual browser smoke check on desktop width
- manual browser smoke check at a mobile breakpoint
- manual verification in both light and dark themes if theme switching is available on the page

## Risks

- the current dashboard JS is organized as a single file, so new logic must stay tidy and avoid turning `renderDashboard` into a dumping ground
- summary language can become misleading if sparse-data cases are not handled explicitly
- aggressive chart restyling could reduce readability if contrast is pushed too low

## Decisions Locked

- no backend changes
- no API shape changes
- executive direction, not analytical or operational
- derived client-side metrics are acceptable and preferred
